# 编排中心开发指南

## 1. 特性介绍

编排中心是一个面向多智能体（Agent）协作的可视化编排平台，支持通过图形化工作流设计器定义 Agent 之间的调用关系与执行流程。后端基于 Python 框架解析流程并驱动 Agent 协同工作。

本特性允许用户为不同操作定义自定义实现，同时为常见操作提供默认实现。通过统一的抽象基类和处理器注册机制，开发者可以扩展系统功能，无需修改核心代码。
除上述自定义实现以外，本特性还提供了可扩展的大模型（LLM）接入框架。LLM 模块采用配置驱动架构，接入新模型通过编辑 `common/config/llm_config.json` 完成，无需编写 Python 代码。

## 2. 约束与限制

- **操作限制**：
  - 自定义处理器注册时应在应用启动时完成，避免在业务处理过程中注册
  - LLM 模块采用配置驱动架构，接入新模型通过编辑 `common/config/llm_config.json` 完成，无需编写 Python 代码（仅新增认证策略时需要少量代码）

## 3. 环境要求

- Python >= 3.10+

### 3.1 搭建环境

该模块设计为Python项目的组成部分。确保以下文件在项目目录中：

**自定义处理器模块：**
- `default_handle.py` - 包含基类、注册表和默认处理器
- `custom_handle.py` - 自定义处理器扩展模块（如需要）

**LLM 模块：**
- `common/llm/__init__.py` - 模块导出文件
- `common/llm/llm.py` - 统一工厂函数
- `common/llm/config/config_reader.py` - JSON 配置读取
- `common/llm/config/llm_config.py` - ModelConfig dataclass
- `common/llm/provider/auth_strategies.py` - Header 签名策略
- `common/llm/provider/generic_llm.py` - 通用 LLM Provider
- `common/config/llm_config.json` - LLM 配置文件

## 4. 使用场景

### 4.1 自定义处理器使用说明

#### 4.1.1 功能介绍

**核心能力：**
- **抽象基类**：为所有处理器定义统一接口
- **默认实现**：为常见操作提供内置处理器
- **自定义扩展**：支持用户注册自定义处理器

**系统架构：**
系统采用注册表模式，核心组件包括：

| 组件 | 职责 |
|------|------|
| BaseHandle | 定义所有处理器的统一抽象接口 |
| HandlerRegistry | 管理处理器注册和获取，提供默认实现兜底 |
| InterfaceType | 定义支持的接口类型枚举 |
| 默认处理器 | 为常见操作提供内置实现 |
| 自定义处理器 | 用户扩展实现，覆盖默认行为 |

```text
┌─────────────────┐
│   业务调用方     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ HandlerRegistry │◄──── 注册自定义处理器
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   BaseHandle    │
│   (抽象基类)     │
└────────┬────────┘
     ┌───┴───┬───────────────┐
     ▼       ▼               ▼
┌───────┐ ┌───────┐  ┌───────────┐
│ 默认  │  │ 默认 │   │  自定义   │
│处理器1│  │处理器2│  │  处理器3  │
└───────┘ └───────┘  └───────────┘
```

#### 4.1.2 何时需要自定义处理器

以下场景需要用户自行实现自定义处理器：

| 场景 | 说明 | 示例 |
|------|------|------|
| 数据库持久化 | 当 `persistence_mode` 配置为 `postgresql` 时，默认的文件存储处理器无法满足需求，需自定义处理器对接数据库存储 | 使用 PostgreSQL 存储 PSOP 和执行记录 |
| 自定义存储介质 | 当需要使用其他存储介质（如 MySQL、MongoDB、Redis 等）时，需自定义处理器实现对应的存储逻辑 | 使用 MySQL 作为 PSOP 存储后端 |
| 存储逻辑定制 | 当需要在保存/查询/删除操作中添加额外业务逻辑时，需自定义处理器扩展默认行为 | 在保存 PSOP 时添加审计日志记录 |

**配置说明：**

在 `etc/conf/server.conf` 中配置 `persistence_mode` 参数：

```properties
persistence_mode=file  # 使用默认文件存储，无需自定义处理器
persistence_mode=postgresql  # 使用数据库存储，需自定义处理器
```

当 `persistence_mode` 设置为非 `file` 值时，系统将优先使用用户注册的自定义处理器。

#### 4.1.3 开发步骤
步骤1：引入所需模块
```python
from common.custom.default_handle import BaseHandler, HandlerRegistry, InterfaceType
```

步骤2：创建自定义处理器

创建继承自BaseHandle的自定义类，并实现handle方法：
```python
class MyCustomHandle(BaseHandle):
    """自定义处理器示例"""
    
    def handle(self, *args, **kwargs):
        """
        自定义处理逻辑
        
        Args:
            *args: 位置参数
            **kwargs: 关键字参数
            
        Returns:
            处理结果
        """
        # 自定义实现
        # 建议：在这里添加具体的业务逻辑
        # 注意：确保方法是异步的
        result = "自定义结果"
        return result
```
步骤3：注册自定义处理器

在应用启动时注册自定义处理器：
```python
# 注册自定义处理器
# 注意：注册操作应在业务处理开始前完成
HandlerRegistry.register(InterfaceType.QUERY, MyCustomHandle)
```
警告：后注册的处理器会覆盖同类型的先前处理器。请确保注册顺序符合预期。

步骤4：使用处理器

在业务代码中获取并使用处理器：
```python
# 获取处理器实例
# HandlerRegistry会自动返回已注册的自定义处理器或默认实现
handle = HandlerRegistry.get_handler(InterfaceType.QUERY)

# 使用处理器执行业务
result = handle.handle(...)
```
步骤5：完整示例

以下是一个完整的自定义处理器实现示例：
```python
from common.custom.default_handle import BaseHandler, HandlerRegistry, InterfaceType

class SaveCustomHandle(BaseHandle):
    """自定义查询处理器"""
    
    def handle(self, query_params=None):
        """
        自定义查询逻辑
        
        Args:
            query_params: 查询参数字典
            
        Returns:
            查询结果列表
        """
        # 示例：实现自定义查询逻辑
        if not query_params:
            return []
        
        # 自定义查询实现
        results = []
        # ... 业务逻辑
        return results

# 注册
HandlerRegistry.register(InterfaceType.SAVE_PSOP, SaveCustomHandle)

# 使用
handler = HandlerRegistry.get_handler(InterfaceType.SAVE_PSOP)
data = handler.handle({"id": 1})
print(data)
```

### 4.2 LLM 模块配置说明

#### 4.2.1 功能介绍

`common/llm` 是统一的 LLM 调用模块，支持 Chat（文本生成）、Embedding（文本向量化）、Reranker（结果重排序）三类模型能力。整个模块采用配置驱动架构，接入新模型通过编辑 JSON 配置文件完成，无需编写 Python 代码。

**核心设计：**

| 组件 | 职责 |
|------|------|
| `ModelConfig` | 从 `llm_config.json` 加载模型配置的 dataclass |
| `GenericLLM` | 唯一的通用 Provider 类，承载 ask_llm / embed / rerank 全部接口 |
| `AUTH_STRATEGIES` | 可插拔的 Header 签名策略注册表，内置 `"aoc_signed"` |
| `get_llm_instance()` / `get_embed_instance()` / `get_rerank_instance()` | 按能力（capability）获取单例实例 |

**目录结构：**
```text
common/llm/
├── __init__.py
├── llm.py                        # 统一工厂函数
├── config/
│   ├── config_reader.py          # JSON 文件读取
│   └── llm_config.py             # ModelConfig dataclass
└── provider/
    ├── auth_strategies.py        # Header 签名策略
    └── generic_llm.py            # 唯一通用 Provider
```

**调用方 API：**
```python
from common.llm import get_llm_instance, get_embed_instance, get_rerank_instance

# Chat — 默认的 get_llm_instance() 等同于 get_llm_instance("chat")
llm = get_llm_instance()
reasoning, answer = llm.ask_llm("今天天气怎么样？")

# Embedding
emb = get_embed_instance()
vector = emb.embed("一段文本")  # 返回 List[float]

# Rerank
rerank = get_rerank_instance()
results = rerank.rerank("查询", ["候选1", "候选2"])  # 返回 List[dict]
```

> **注意：** 实例是单例缓存的，多次调用返回同一个对象。

#### 4.2.2 配置文件格式

配置文件位于 `common/config/llm_config.json`，顶层按能力（capability）分组，每个能力块配置一个模型：

```json
{
  "chat":   { ... },
  "embed":  { ... },
  "rerank": { ... }
}
```

##### 通用字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `description` | `string` | 否 | 模型描述，用于日志 |
| `model` | `string` | 否 | 模型名称，通过 `$MODEL` 占位符注入请求体 |
| `url` | `string` | **是** | 模型 API 端点地址 |
| `api_key` | `string` | 否 | API 密钥，auth 为 null 时自动作为 `Authorization: Bearer` 头 |
| `enable_thinking` | `boolean` | 否 | 思考模式开关，通过 `$ENABLE_THINKING` 占位符注入 |
| `auth` | `object/string/null` | 否 | 认证策略，见下文 |
| `headers` | `object` | 否 | 额外的静态 HTTP 头，合并到最终请求头 |
| `body` | `object` | **是** | 请求体模板，支持占位符 |
| `response` | `object` | **是** | 响应提取路径 |

##### 认证策略（`auth` 字段）

| 策略名 | 形式 | 说明 |
|--------|------|------|
| `null` | `"auth": null` | 无特殊认证，`api_key` 非空时自动加 Bearer 头 |
| `"aoc_signed"` | 对象形式（见示例） | AOC 平台签名 Header（`x-sg-*` 系列） |

`aoc_signed` 对象形式的必填参数：

| 参数 | 说明 |
|------|------|
| `app_key` | AOC 应用密钥 |
| `app_secret` | AOC 应用密钥 Secret |
| `authorization` | Bearer 令牌头 |
| `api_code` | API 编码 |

可选参数有默认值：`scenario_code` ("B99999999999")、`scenario_version` ("V1")、`ability_code` ("A999999999")、`api_version` ("1.0")、`test_flag` ("1")。

##### 请求体占位符

`body` 是 JSON 对象模板，支持以下占位符：

| 占位符 | 展开为 | 适用能力 |
|--------|--------|----------|
| `$MODEL` | `model` 字段值 | chat, embed, rerank |
| `$PROMPT` | `ask_llm()` / `embed()` 的 prompt 参数 | chat, embed |
| `$QUERY` | `rerank()` 的 query 参数 | rerank |
| `$DOCUMENTS` | `rerank()` 的 documents 参数（JSON 数组） | rerank |
| `$ENABLE_THINKING` | `enable_thinking` 字段值 | chat, embed, rerank |

> 值的完整替换规则：若字符串值完全等于占位符（如 `"$MODEL"`），替换为原类型（bool→`true`/`false`，list→JSON 数组）；若占位符只是子串（如 `"prefix-$MODEL"`），替换为字符串。

##### 响应提取路径（`response` 字段）

定义从 API 返回 JSON 中提取数据的点分路径：

| 能力 | response 键 | 说明 |
|------|-------------|------|
| chat | `answer` | 回答文本提取路径 |
| chat | `reasoning` | 推理/思考过程提取路径（可选） |
| embed | `embedding` | 向量数组提取路径 |
| rerank | `results` | 重排结果提取路径 |

路径语法：`.` 分隔字段名，数字为数组索引。例如：
- `"choices.0.message.content"` → `data['choices'][0]['message']['content']`
- `"data.0.embedding"` → `data['data'][0]['embedding']`
- `"results"` → `data['results']`

#### 4.2.4 何时需要写代码

常规场景编辑 JSON 配置即可接入。仅在以下情况需要扩展：

| 场景 | 做法 | 说明 |
|------|------|------|
| 新增认证策略 | 在 `provider/auth_strategies.py` 注册函数 | 函数签名 `(params: dict) -> dict[str, str]` |
| 模型的 response 格式非常特殊 | 很少见，response 路径已覆盖 OpenAI 和主流格式 | response 路径支持任意嵌套点分路径 |

> 早期版本需要为每个新模型写一个 Python 类（继承 `BaseLLM`，实现 `_build_request_body`/`_parse_response`）。**现已废弃，请不要使用旧方式。**

#### 4.2.5 从旧配置迁移

旧配置使用 `LLMType` 枚举区分模型，如 `"aoc_chat_llm"` / `"openai_style_llm"`。新架构按能力命名：

| 旧 key | 新 key | 附加变化 |
|--------|--------|----------|
| `openai_style_llm` | `chat` | `api` → `url` |
| `aoc_chat_llm` | `chat` | `extra.xxx` → `auth.xxx`；`request_template`（JSON 字符串）→ `body`（JSON 对象） |
| `aoc_embedding_llm` | `embed` | 同上 |
| `aoc_reranker_llm` | `rerank` | 同上 |

占位符语法变化：`{prompt}` → `$PROMPT`，`{enable_thinking}` → `$ENABLE_THINKING`，`{query}` → `$QUERY`，`{documents}` → `$DOCUMENTS`。

## 5. FAQ

### 5.1 自定义处理器未生效
**现象描述**：注册了自定义处理器，但调用时仍使用默认处理器。

**可能原因**：
- 1.注册操作在获取处理器之后执行
- 2.注册的接口类型与实际使用的不匹配
- 3.自定义处理器未正确继承BaseHandle

**解决办法**：
- 1.确保在应用启动时、任何业务处理之前完成注册
- 2.检查注册和获取时使用的InterfaceType是否一致
- 3.确认自定义处理器正确继承了BaseHandle并实现了handle方法

### 5.2 新模型配置不生效？

**现象描述**：修改了 `llm_config.json` 但没有生效。

**可能原因**：
- 1. JSON 格式错误（缺少逗号、括号不匹配等）
- 2. 能力 key 名称不正确（应为 `chat`、`embed`、`rerank`）
- 3. 服务启动后才修改配置文件（配置在模块首次导入时加载）
- 4. `body` 字段的占位符变量名写错（应为大写，如 `$PROMPT` 而非 `$prompt`）

**解决办法**：
- 1. 使用 JSON 校验工具检查配置文件语法
- 2. 确认顶层 key 为 `"chat"`、`"embed"`、`"rerank"`
- 3. 修改配置后重启服务，使其重新加载
- 4. 检查 `response` 路径是否匹配实际 API 返回格式

### 5.3 模型调用报错 "Unable to extract..."

**现象描述**：调用 `embed()` 或 `rerank()` 时报 `ValueError: Unable to extract ...`

**可能原因**：`response` 配置的提取路径与实际 API 返回的 JSON 结构不匹配。

**解决办法**：
- 1. 先用 `curl` 或 Postman 调一次该 API，获取真实的响应 JSON
- 2. 根据实际 JSON 结构调整 `response` 路径，如 `"data.0.embedding"` 或 `"data[0].embedding"` 均可
- 3. 确认路径中的数组索引和字段名区分大小写
