# 编排中心开发指南

## 1. 特性介绍
编排中心是一个面向多智能体（Agent）协作的可视化编排平台，支持通过图形化工作流设计器定义 Agent 之间的调用关系与执行流程。后端基于 Python 框架解析流程并驱动 Agent 协同工作，帮助用户高效构建、管理和运行复杂的 Agent 协作流程。

本特性允许用户为不同操作定义自定义实现，同时为常见操作提供默认实现。通过统一的抽象基类和处理器注册机制，开发者可以灵活扩展系统功能，无需修改核心代码。
除上述自定义实现以外本特性还提供了一个可扩展的大模型（LLM）接入框架，支持用户自定义实现不同厂商或类型的大模型。框架基于注册器模式，可以方便地添加新的大模型类型并在代码中使用。

## 2. 约束与限制
- **操作限制**：
  - 自定义处理器注册时应在应用启动时完成，避免在业务处理过程中注册
  - 自定义LLM实现应在应用启动前完成，避免应用启动后完成导致找不到自定义的LLM

## 3. 环境要求
- Python >= 3.10+

## 4. 使用场景
### 4.1 自定义处理器使用说明
#### 4.1.1 功能介绍
**核心能力：**
- **抽象基类**：为所有处理器定义统一接口
- **默认实现**：为常见操作提供内置处理器
- **自定义扩展**：支持用户注册自定义处理器

**系统架构：**
系统采用注册表模式，核心组件包括：

| 组件 | 职责 |
|------|------|
| BaseHandle | 定义所有处理器的统一抽象接口 |
| HandlerRegistry | 管理处理器注册和获取，提供默认实现兜底 |
| InterfaceType | 定义支持的接口类型枚举 |
| 默认处理器 | 为常见操作提供内置实现 |
| 自定义处理器 | 用户扩展实现，覆盖默认行为 |

```
┌─────────────────┐
│   业务调用方     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ HandlerRegistry │◄──── 注册自定义处理器
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   BaseHandle    │
│   (抽象基类)     │
└────────┬────────┘
     ┌───┴───┬───────────────┐
     ▼       ▼               ▼
┌───────┐ ┌───────┐  ┌───────────┐
│ 默认  │  │ 默认 │   │  自定义   │
│处理器1│  │处理器2│  │  处理器3  │
└───────┘ └───────┘  └───────────┘
```
#### 4.1.2 开发步骤
步骤1：引入所需模块
```python
from default_handle import BaseHandle, HandlerRegistry, InterfaceType
```

步骤2：创建自定义处理器

创建继承自BaseHandle的自定义类，并实现handle方法：
```python
class MyCustomHandle(BaseHandle):
    """自定义处理器示例"""
    
    def handle(self, *args, **kwargs):
        """
        自定义处理逻辑
        
        Args:
            *args: 位置参数
            **kwargs: 关键字参数
            
        Returns:
            处理结果
        """
        # 自定义实现
        # 建议：在这里添加具体的业务逻辑
        # 注意：确保方法是异步的
        result = "自定义结果"
        return result
```
步骤3：注册自定义处理器

在应用启动时注册自定义处理器：
```python
# 注册自定义处理器
# 注意：注册操作应在业务处理开始前完成
HandlerRegistry.register(InterfaceType.QUERY, MyCustomHandle)
```
警告：后注册的处理器会覆盖同类型的先前处理器。请确保注册顺序符合预期。

步骤4：使用处理器

在业务代码中获取并使用处理器：
```python
# 获取处理器实例
# HandlerRegistry会自动返回已注册的自定义处理器或默认实现
handle = HandlerRegistry.get_handler(InterfaceType.QUERY)

# 使用处理器执行业务
result = handle.handle(...)
```
步骤5：完整示例

以下是一个完整的自定义处理器实现示例：
```python
from default_handle import BaseHandle, HandlerRegistry, InterfaceType

class SaveCustomHandle(BaseHandle):
    """自定义查询处理器"""
    
    def handle(self, query_params=None):
        """
        自定义查询逻辑
        
        Args:
            query_params: 查询参数字典
            
        Returns:
            查询结果列表
        """
        # 示例：实现自定义查询逻辑
        if not query_params:
            return []
        
        # 自定义查询实现
        results = []
        # ... 业务逻辑
        return results

# 注册
HandlerRegistry.register(InterfaceType.SAVE_PSOP, SaveCustomHandle)

# 使用
handler = HandlerRegistry.get_handler(InterfaceType.SAVE_PSOP)
data = handler.handle({"id": 1})
print(data)
```
#### 4.1.3 默认处理器说明

若未注册自定义处理器，系统将使用以下默认实现：

| 处理器 | 对应接口类型 | 功能描述 | 参数说明 |
|--------|-------------|----------|----------|
| SavePsopHandler | SAVE_PSOP | 保存PSOP工作流 | psop: PSOP对象，待保存的工作流 |
| GetAllPsopsHandler | GET_ALL_PSOP | 查询所有PSOP列表 | 无参数 |

| DeletePsopHandler | DELETE_PSOP | 删除指定PSOP | workflow_id: str，工作流唯一标识 |
| SaveExecutionRecordHandler | SAVE_EXECUTION_RECORD | 保存执行记录 | record: ExecutionRecord对象，待保存的执行记录 |
| ListExecutionRecordsHandler | LIST_EXECUTION_RECORDS | 查询所有执行记录列表 | 无参数 |

| DeleteExecutionRecordHandler | DELETE_EXECUTION_RECORD | 删除指定执行记录 | execution_id: str，执行记录唯一标识 |

### 4.2 自定义LLM使用说明
#### 4.2.1 功能介绍
**核心能力：**
- **抽象基类**：为所有LLM实现定义统一接口（`BaseLLM`）
- **注册器机制**：通过装饰器模式管理LLM提供者的注册与获取
- **配置驱动**：基于JSON配置文件管理不同LLM的连接参数
- **实例缓存**：单例模式管理LLM实例，避免重复创建

**系统架构：**
系统采用注册器模式与装饰器模式，核心组件包括：

| 组件 | 职责 |
|------|------|
| BaseLLM | 定义所有LLM的抽象基类，提供统一的调用接口 |
| LLMProviderRegistry | 管理LLM提供者注册，支持装饰器注册 |
| LLMType | 定义支持的LLM类型枚举 |
| LLMConfig/LLMConfigItem | 管理LLM配置信息（模型、API、密钥等） |
| 默认LLM实现 | OpenAIStyleLLM等内置实现 |
| 自定义LLM | 用户扩展实现，支持任意厂商接入 |

```
┌─────────────────────┐
│     业务调用方       │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   get_llm_instance  │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ LLMProviderRegistry │◄──── @registry_provider 注册自定义LLM
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│      BaseLLM        │
│     (抽象基类)       │
└──────────┬──────────┘
     ┌──────┴──────┬───────────────┐
     ▼             ▼               ▼
┌─────────┐ ┌──────────┐  ┌─────────────┐

│StyleLLM │ │ ChatLLM  │  │  MyLLM      │
└─────────┘ └──────────┘  └─────────────┘
```

```python
from typing import Union, Tuple
from common.llm.config.llm_config import LLMType, LLMConfig
from common.llm.provider.base_llm import BaseLLM
from common.llm.provider.llm_provider_registry import registry_provider

@registry_provider(LLMType.MY_CUSTOM_LLM)
class MyCustomLLM(BaseLLM):
    
    def __init__(self, llm_config: LLMConfig):
        super().__init__(llm_config)
        # 初始化自定义客户端
        self.client = {} # 按实际情况实现
    def _ask_llm(self, prompt: str) -> Union[str, Tuple[str, str]]:
        # 调用自定义API, 仅作为示例说明，具体实现按实际情况进行修改
        response = self.client.generate(
            model=self.model,
            prompt=prompt,
            enable_thinking=self.enable_thinking
        )
        
        # 根据API返回格式处理
        if self.enable_thinking and hasattr(response, 'reasoning'):
            reasoning = response.reasoning
            answer = response.content
            return reasoning, answer
        else:
            return response.content
```

## 5. FAQ
### 5.1 自定义处理器未生效
**现象描述**：注册了自定义处理器，但调用时仍使用默认处理器。

**可能原因**：
- 1.注册操作在获取处理器之后执行
- 2.注册的接口类型与实际使用的不匹配
- 3.自定义处理器未正确继承BaseHandle

**解决办法**：
- 1.确保在应用启动时、任何业务处理之前完成注册
- 2.检查注册和获取时使用的InterfaceType是否一致
- 3.确认自定义处理器正确继承了BaseHandle并实现了handle方法

### 5.2 新添加的大模型没有被注册怎么办？
**现象描述**：已经按照文档添加了自定义大模型，但在调用时提示找不到该类型，或仍然使用默认的大模型。

**可能原因**：
- 1.未正确使用 @registry_provider 装饰器
- 2.自定义大模型类未在 __init__.py 中被导入
- 3.LLMType 枚举的字符串值与配置文件中的 key 不一致
- 4.自定义大模型文件未被 Python 解释器加载 

**解决办法**：
- 1.确认在自定义类定义处使用了 @registry_provider(LLMType.YOUR_TYPE) 装饰器
- 2.在 llm/__init__.py 中添加导入语句：from .your_custom_llm import YourCustomLLM
- 3.检查 LLMType 枚举值与 llm_config.json 中的配置键名是否完全一致（包括大小写）
- 4.确保自定义大模型模块在应用启动时被导入，可在 __init__.py 中导入，或在主程序入口导入
